Here is the detailed technical architecture for the Simple Budget Tracker application, designed for direct implementation by a Code-generation AI.

---

## 1. Architecture Overview

The system is a monolithic web application built on Next.js 14 using the App Router. It follows a server-centric pattern where the frontend and backend are tightly integrated within the same project.

-   **Frontend:** The UI is composed of React Server Components (RSCs) for rendering static and data-driven content, and Client Components for interactivity. This minimizes the client-side JavaScript bundle size.
-   **API/Mutations:** Data fetching for pages is primarily done within RSCs directly on the server. Data mutations (Create, Update, Delete) are handled by Next.js Server Actions, which are secure RPC-like functions callable from Client Components. This co-locates mutation logic with the components that use it. A few GET-only API routes will be used for client-side data fetching where necessary (e.g., dynamic lookups).
-   **Database:** A single PostgreSQL database serves as the source of truth. The application interacts with it exclusively through the Prisma ORM, ensuring type safety between the database and application code.
-   **Authentication:** User authentication is managed by NextAuth.js, which handles session creation, verification, and management via secure HTTP-only cookies.

## 2. Technology Stack

| Layer      | Technology                                    | Justification                                                                                                                                                             |
| :--------- | :-------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Frontend** | Next.js 14 (App Router) + React 18 + TypeScript | Provides a robust framework with Server Components for performance, a clear project structure, and static typing for maintainability and fewer runtime errors.                |
| **Styling**  | Tailwind CSS + shadcn/ui                      | Utility-first CSS for rapid, consistent styling. shadcn/ui provides accessible, unstyled component primitives that can be easily customized and owned by the application. |
| **API**      | Next.js Server Actions & Route Handlers       | Server Actions are used for form submissions and mutations (CUD), simplifying client-side state management. Route Handlers are used for read-only (R) data endpoints.      |
| **Database** | PostgreSQL with Prisma ORM                    | PostgreSQL is a powerful, open-source relational database. Prisma offers a type-safe ORM that simplifies database access and migrations.                                     |
| **Auth**     | NextAuth.js (v5)                              | The standard for Next.js authentication. Simplifies complex auth flows (credentials, OAuth), session management, and security, integrating seamlessly with the App Router. |
| **Deployment** | Docker + Google Cloud Run                     | Containerization with Docker ensures consistency across environments. Cloud Run provides a scalable, serverless, and cost-effective platform for deploying containers.      |

## 4. Database Schema

**File:** `prisma/schema.prisma`

```prisma
// This is your Prisma schema file,
// learn more about it in the docs: https://pris.ly/d/prisma-schema

generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model User {
  id            String    @id @default(cuid())
  name          String?
  email         String?   @unique
  emailVerified DateTime?
  image         String?
  password      String? // For credentials provider
  accounts      Account[]
  sessions      Session[]
  transactions  Transaction[]
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt
}

model Account {
  id                String  @id @default(cuid())
  userId            String
  type              String
  provider          String
  providerAccountId String
  refresh_token     String? @db.Text
  access_token      String? @db.Text
  expires_at        Int?
  token_type        String?
  scope             String?
  id_token          String? @db.Text
  session_state     String?

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([provider, providerAccountId])
}

model Session {
  id           String   @id @default(cuid())
  sessionToken String   @unique
  userId       String
  expires      DateTime
  user         User     @relation(fields: [userId], references: [id], onDelete: Cascade)
}

model VerificationToken {
  identifier String
  token      String   @unique
  expires    DateTime

  @@unique([identifier, token])
}

enum TransactionType {
  INCOME
  EXPENSE
}

model Transaction {
  id          String          @id @default(cuid())
  description String
  amount      Decimal         @db.Decimal(10, 2)
  type        TransactionType
  date        DateTime

  userId String
  user   User   @relation(fields: [userId], references: [id], onDelete: Cascade)

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@index([userId, date])
}
```
